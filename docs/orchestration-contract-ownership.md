### Action Ownership

#### 1. Ticket Create

- Single writer of record: `packages/core/src/tickets/create.ts:createTicket` owns id allocation, collision checks, open-ticket file creation, `INDEX.md` insertion, `order.json` append, and round-trip validation.
- Format owner: `packages/core/src/tickets/compose.ts:composeTicketMarkdown`; loader/id owner: `packages/core/src/tickets/loader.ts:nextTicketId`; INDEX row owner: `packages/core/src/tickets/index-helpers.ts:insertOpenTicketIndexRow`.
- Transporters: daemon route `packages/daemon/src/routes.ts:createTicket`; CLI wrapper `packages/cli/src/create-ticket.ts:createTicketViaCli`; CLI live-daemon guard in `packages/cli/src/run.ts`; Deb escalation path `packages/core/src/runner/triage.ts:createDebEscalationTicket`; optional authoring Play dispatch through `packages/daemon/src/launcher.ts:requestAuthoringPlay`.
- Second-copy contract: `packages/personas/base/plays/create-ticket.md` repeats next-id, filename slug, frontmatter, and INDEX-row instructions, and still says only the ticket file plus `INDEX.md` should change. Live code also writes `cocoder/tickets/order.json`, so the Play is a format/order duplicate and currently weaker than the core helper.
- Edge cases owned by core/tests: explicit id collision across paths, INDEX, order, and loaded tickets; malformed/missing order recovery; Markdown table escaping; generated slug filename; round-trip parse after write.
- Pinning tests: `packages/core/tests/tickets.test.ts` pins create/id/order/collision behavior; `packages/daemon/tests/authoring-play.test.ts` pins create-ticket Play dispatch/commit shape; `packages/cli/tests/create-ticket.test.ts` and `packages/cli/tests/oz-args.test.ts` pin CLI creation and argument parsing.
- UNPINNED: create-ticket Play tests do not pin the core `order.json` append; no Oz chat/tool create-ticket path is pinned because Oz currently has no create-ticket authoring tool.

#### 2. Ticket Close

- Single writer of record: `packages/core/src/tickets/close.ts:closeTicket` owns open-to-closed move, status mutation, resolution append, INDEX move, order pruning, stale-order cleanup, and closed-ticket validation.
- Transporters: auto-close after successful ticket runs in `packages/daemon/src/launcher.ts:closeTicketAfterSuccessfulRun`; in-run verify-gated close in `packages/core/src/runner/agent-step.ts:handleVerifiedTicketClose`, triggered by optional verify `ticketClose` parsed in `packages/core/src/runner/io.ts`, after a `pass` verdict and the atom's own commit gate, then committed through shared `runCommitGate` with `cocoder/tickets/**` plus `commitOnlyScope` so `recordCommitLink` ledgers it and wrap audit does not flag it; active-run async dashboard/founder close in `packages/daemon/src/authoring-queue.ts:enqueueAuthoring` / `packages/daemon/src/authoring-queue.ts:drainAuthoringQueue` for `action:'ticket-close'`; reconciliation close in `packages/daemon/src/launcher.ts:requestReconciliationClose`; close-confirmation action in `packages/daemon/src/launcher.ts:requestTicketCloseConfirmation` (with bounded recovery when gate refuses `not-awaiting-ticket-close-confirmation` but the run's delivered closeout authoritatively requests close); daemon run-action projection in `packages/daemon/src/routes.ts:runActions`; CLI wrappers `packages/cli/src/close-ticket.ts:closeTicketViaCli` and `packages/cli/src/client.ts:confirmTicketCloseViaDaemon` (`cocoder oz confirm-ticket-close <runId>`); Oz chat reconciliation command and `confirm-close <runId>` in `packages/daemon/src/oz-chat.ts`.
- Close gate: `packages/daemon/src/ticket-close-gate.ts` owns run-state evidence for unattended daemon close lanes. Reconciliation and queued closes refuse when the latest run for that ticket is parked on an unanswered founder decision; explicit close confirmation is allowed only when the same run's validated wrap event carries `ticketCloseDecision: "ask"`, emitted by `packages/core/src/runner/runner.ts` from the delivered closeout (`packages/core/src/plays/founder-closeout.ts:deriveTicketCloseDecision`, including `Run Status:` label normalization). The `cocoder oz close-ticket` command uses the reconciliation daemon route when Oz is live; `cocoder oz confirm-ticket-close` requires a live daemon (awaiting-founder state is in memory) and routes to close confirmation, not reconciliation. Daemonless `closeTicketViaCli` is an operator-only filesystem spine with no run-store evidence.
- Second-copy contract: no second Markdown writer was found. Close confirmation text and pending-close state are projected by daemon/UI surfaces, but file mutation stays in the core helper.
- Edge cases owned by core/tests: already closed, missing open ticket, missing/malformed order, stale order-only reconciliation, closed ticket parse after move, pending-close run finalization, active-run refusal for the same ticket, in-run `priority-mismatch` skip when the requested open ticket belongs to neither the active priority nor the run's own `ticketId`, in-run `verify-fail` skip when `ticketClose` rides a failed verdict, and loud `in-run-ticket-close-commit-failed` failure instead of silently stranding a half-written close.
- Pinning tests: `packages/core/tests/tickets.test.ts` pins close/prune behavior; `packages/core/tests/runner.test.ts` pins in-run verify-gated close (pass closes through gate, fail skips, stale-order reconciliation is ledgered, priority mismatch skips, commit failure fails loudly) and wrap `needs closing` → `ticketCloseDecision: ask`; `packages/core/tests/io.test.ts` pins verify-artifact `ticketClose` parse and malformed rejection; `packages/daemon/tests/mutations.test.ts` pins auto-close, deferred confirmation, active-run refusal, stale order cleanup, pending run finalization, route-level refusal while the latest ticket run awaits founder input, queued-close drain refusal after an awaiting-founder wrap, and stranded-`none` recovery through close confirmation; `packages/daemon/tests/reconciliation-close-op.test.ts` pins Oz reconciliation close and the no-mutation awaiting-founder refusal; `packages/daemon/tests/oz-chat.test.ts` pins `confirm-close` dispatch; `packages/cli/tests/close-ticket.test.ts` pins CLI close/commit behavior; `packages/cli/tests/client.test.ts` pins `confirm-ticket-close` daemon POST and 409 refusal surfacing.
- UNPINNED: no gap found for the file mutation path; UI pending-close display is pinned separately from the writer and does not own mutation semantics.

### Active-Run Queue And Held Mutations

- **Deterministic governance authoring queue:** `packages/daemon/src/authoring-queue.ts` accepts ticket create/close/repoint/reorder and priority-create while `ctx.inFlight` holds the workspace. Routes enqueue with HTTP 202; `drainAuthoringQueue` commits at the per-atom safe seam (`packages/daemon/src/launcher.ts` post-atom hook) and as a wrap-time backstop. Commits are ledgered so wrap audit does not flag them as out-of-band. Pending entries surface through `listQueuedAuthoring` / `listTickets`.
- `packages/daemon/src/launcher.ts:launchRun` refuses a second active run in the same workspace and reserves a pending entry before launch, closing the concurrent POST race.
- `packages/daemon/src/launcher.ts:requestAuthoringPlay` still refuses model-mediated authoring Play dispatch while another run is actively `running` on the workspace; post-wrap continuation of the same run is allowed.
- `packages/daemon/src/launcher.ts:requestSupportCommitRun` refuses support commits while the target run is still running or another run owns the workspace, and separately refuses raw archive support commits after wrap.
- Reconciliation close/repoint, archive confirmation, ticket close confirmation, workspace delete, daemon restart, Oz repair/action, and `drainDaemonReload` retain their prior active-run refusal or hold behavior.
- `packages/cli/src/run.ts` refuses loop-down `close-ticket` and `create-ticket` when the daemon is live (routes to daemon-backed path); during an active run those daemon paths enqueue instead of refusing.

`wrap-up.md` owns the closeout **format**, but the runner's `WRAP-UP READY` artifact owns the visible

founder closeout directly. The enforcer now pins that separation by allowing exactly one delivery

live persona/status surfaces.

Disposition (run_148): **considered and deliberately not actioned** (ticket `0018`, closed). The
triggering commits (`90599db`, `aa7addc`) were correct, green, and founder-kept — not a correctness

commit-withholding — the ADR-0023 / F21 anti-pattern the rebuilds removed. A detection-only version is
governance-of-governance (F5). No guard is warranted.

says machinery code is verified before the spine commits and that the receipt is derived from the spine's
actual result (`cocoder/decisions/0023-workspace-commit-spine.md:115-128`,
`cocoder/decisions/0023-workspace-commit-spine.md:150-156`). The observed commit `90599db83d5a701cb88b80b290bc3a4c3b910afe`
is on `main`, is authored as a direct git commit, and touched `packages/core/src/runner/runner.ts`,
`packages/core/tests/runner.test.ts`, and `docs/founder-brief-format-durability.md` (`git show --stat
--oneline --name-only 90599db`; `git branch --contains 90599db`). The durable run_145 commit record lists
the runner-mediated commits as `80f496f`, `0b1d5a5`, and `661f97e`, not `90599db`
(`cocoder/runs/7-run_145/commits.jsonl:1-3`).

This fits the class because a contract existed in governance and prompt text, but a path still let
machinery code reach the branch without the runner's verify/receipt surface. The founder call is whether
this priority should add an enforcement atom now, because a true guard may touch git workflow, commit-spine
policy, or host-side controls rather than only prompt/runtime text.

Per-axis verdicts:

- **Trigger class: real for `lifecycle-triggered` vs `persona-requested`; suspect for `tool/API-triggered`.**
  Collapsing lifecycle and persona request would lose the mandatory-run-wrap registry and the request-lane
  rejection that keeps `wrap-up` out of persona discretion. The unshipped `tool/API-triggered` value is
  reserved/forward-declared by ADR-0028 for API-triggered dispatch.
- **Execution model: real.** Collapsing prompt-only and hybrid would lose deterministic precheck gating:
  a failed precheck currently prevents any model invocation, while a successful one feeds evidence into
  the prompt.
- **Write authority: real as `writeScope`; suspect as `interactive` class language.** Declared
  `writeScope` controls observable commit receipts and scoped authoring commits; the `interactive` kind
  has no shipped Play and is reserved/forward-declared by ADR-0028 for interactive browser control.

**Overall taxonomy verdict: `suspect`.** The observable contract is load-bearing as three fields
(`triggerClass`, `executionModel`, `writeScope`/hosting kind), while a flat list of named classes
overstates the real system: `tool/API-triggered` and `interactive` are modeled but unused, and
prompt-only/hybrid are execution behaviors rather than launch classes. **Disposition:** ADR-0028 replaces
the class-list framing with three axes, reserves `tool/API-triggered` and `interactive` for their named
future runtime paths, and keeps the tests above as the behavior contract.
