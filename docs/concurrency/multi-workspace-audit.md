# Multi-Workspace Daemon State Audit

Audit scope: daemon/core runtime state that can assume one active run. Evidence was gathered from the
live tree with `rg -n "ctx\.inFlight" packages/daemon/src` plus targeted reads of launcher, store,
dashboard, runner, and cmux code. Classifications use exactly:

- LEGITIMATELY GLOBAL
- MUST BE PER-WORKSPACE
- ALREADY PER-RUN/PER-WORKSPACE SAFE

## Findings To Hand To Implementation Atoms

1. `requestOzRepair` and `requestOzAction` are the only audited production sites that clearly block
   cross-workspace work through `ctx.inFlight.size > 0` even though the requested operation is
   workspace-targeted by `workspaceId`.
2. Daemon restart/reload gates that use `ctx.inFlight.size` are legitimately global because they bounce
   the one daemon process and can orphan any active run.
3. The deterministic run launcher already uses `workspaceId` locking end-to-end: reserve pending,
   promote to run id, release on failure/settle, and queue governance edits per workspace.
4. Store, counters, stop controllers, run-driver state, status/read surfaces, and cmux tracking are
   already keyed by workspace/root/run where they need to be. Do not introduce an intra-root hard
   boundary; the hard isolation boundary remains the workspace/root.
