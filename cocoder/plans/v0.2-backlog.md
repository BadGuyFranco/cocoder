# v0.2 Backlog — Consolidated deferred items

**Created:** 2026-05-21 | **Updated:** 2026-05-21
**Type:** Backlog (reference file; not a Playbook)
**Status:** Capture-only — promote items to a priority when ready

> Items deferred out of v0.1 during the Master Playbook + Sub-Playbook A–D scoping pass. This file is a holding pen, not a commitment. When a v0.2 priority opens, items move into `priorities/v0.2-*/plans/`.

## How this file works

- **Append-only during v0.1.** Edit only to add new deferrals discovered while executing Sub-Playbooks A–D.
- **Each item must capture WHY it was deferred** (so future-us doesn't relitigate the same trade-off).
- **No prioritization here.** Ordering happens when v0.2 Witness begins.
- **Not a Playbook.** No WISER phases, no Final Check. When promoted, the receiving Playbook will Witness the relevant rows.

## Deferred items

### From Sub-Playbook A (Foundation)

| Item | Why deferred from v0.1 |
|---|---|
| Full TS migration of `packages/core/.mjs` files | ADR-0004: extract verbatim first to preserve behavior; migrate only after CI is green and a regression baseline exists |
| Keychain / OS-credential integration for secrets | E3.5 — `${env:NAME}` resolver is sufficient for v0.1; native keychain adds platform surface area |
| Multi-tenant install profiles (e.g., `<CoCoder>/local/profiles/{work,personal}/`) | Single profile satisfies the founder use case; profiles add resolver complexity disproportionate to v0.1 demand |
| Sophisticated config merge strategies (e.g., `!override`, `!append` tags) | Deep-merge-objects + replace-arrays is documented and predictable for v0.1 |

### From Sub-Playbook B (Personas + Template)

| Item | Why deferred from v0.1 |
|---|---|
| Phil-style fully custom persona authoring docs + scaffold command | v0.1 ships Bob/Talia/Quinn defaults; custom personas land after the default set is proven |
| Persona prompt fragmentation (public head + private depth split) | Public-only fragments first; the public/private split is a CoBuilder convention we'll port only after v0.1 docs are stable |
| Workspace template variants (web, CLI, library, data) | v0.1 ships one template; variants need at least one external adopter's feedback first |

### From Sub-Playbook C (Oz MVP)

| Item | Why deferred from v0.1 |
|---|---|
| Web-based remote dashboard (non-localhost) | Daemon binds `127.0.0.1` only in v0.1; remote exposure requires a separate threat model |
| Multi-tenant daemon (multiple users on one machine) | Single-user assumption is correct for the target persona (solo builders); revisit when teams adopt |
| Persistent workspace history / run replay | v0.1 surfaces current state; history viewer is an Oz-dashboard feature, not Oz-MVP |
| Oz HTTP API formal contract publication (OpenAPI / typed client) | Schemas live in `packages/schemas/src/oz/` (E3.6 reserves the namespace); formal client generation defers |

### From Sub-Playbook D (Docs / Publish)

| Item | Why deferred from v0.1 |
|---|---|
| Public-facing roadmap doc | v0.1 ships with `cocoder/PRIORITIES.md` as the only public roadmap; a separate `docs/roadmap.md` waits for v0.2 |
| Brand / marketing site | Out of scope for an open-source v0.1; lives in a separate repo when needed |
| Community contribution playbook | `CONTRIBUTING.md` is enough until there are external contributors |

### Cross-cutting

| Item | Why deferred from v0.1 |
|---|---|
| Plugin architecture for third-party personas | Premature; first prove the persona contract with the in-repo set |
| Workspace memory advanced features (graph indexing, cross-workspace federation) | v0.1 ships text-file memory stubs only |
| Telemetry / opt-in usage analytics | Privacy-sensitive; needs an ADR before any code lands |

## Promotion checklist

When opening a v0.2 priority:

1. Copy relevant rows into the new priority's Witness section as "candidate scope"
2. Re-derive the "why deferred" reasoning — if it still holds, the item stays deferred or shrinks
3. Delete promoted rows from this file (kept lean)
