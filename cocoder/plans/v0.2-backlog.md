| **oz-daemon → core boundary: replace direct `.mjs` imports with CLI subprocess invocation per PC-Q4=A extension** | Surfaced 2026-05-23 during Sub-Playbook C Solve Batch 3 (PR #38). `packages/oz-daemon/src/settings.ts` currently uses `// @ts-expect-error` to import `resolveConfig` + `setInstallConfigValue` from `../../core/lib/config.mjs` (cross-package relative path, bypasses the package boundary, silences TS rather than fixing types). Works at runtime; security invariant C-S5 is genuinely proven. But the spirit of PC-Q4=A (daemon doesn't directly couple to core's god modules; subprocess `cocoder launch`/`stop` instead) extends naturally to stateful config operations: daemon should spawn `cocoder config get` / `cocoder config set` subprocesses. `cocoder config get` already defaults to UNRESOLVED secrets per M4.5, so the subprocess hand-off preserves C-S5 byte-for-byte. **Do this refactor when the C Expand dashboard Settings page lands** — one refactor with both sides aware of the new contract beats a v0.1 retroactive cleanup. Pure-functional core imports (e.g., `paths.mjs` token resolvers used in C-S8 registry tests) are NOT in scope for this entry; they're well-typed helpers that don't need the subprocess pattern. The TS type-resolution gap on the `.mjs` boundary (the actual cause of `@ts-expect-error`) is a separate question — possible fixes: emit `.d.ts` from `packages/core` via JSDoc + `tsc`, or define explicit `types` field in core's `package.json`. Coordinate with the broader TS-wrapper-identity decision already in this backlog (per Sub-Playbook F deferral). |
| **Replace tmux with cmux as the orchestration multiplexer; commit CoCoder to macOS-only as the v0.2+ platform stance** | Founder captured 2026-05-23 as a forward-looking architectural intent. cmux is macOS-native; adopting it ends the "macOS-first" softness in ADR-0001 (currently macOS-first but not exclusive) and replaces it with macOS-only. Touches every orchestration callsite (launch.mjs tmux invocations, debugger.mjs concurrency map, multi-workspace socket isolation per ARCHITECTURE.md, Oz's tmux observability from Sub-Playbook C). **ADR-graduating** (commits the platform stance + supersedes ADR-0001's macOS-first scope clause). **Pairs naturally with the launch.mjs decomposition entry above** — the proposed v0.2-architectural priority's `terminal/attach-strategies.mjs` strategy interface IS the seam where cmux would slot in; doing both together prevents thrashing the same files twice. **Suggest opening as a dedicated v0.2 priority** (e.g. `v0.2-multiplexer-migration`) when ready; the priority's Witness should evaluate cmux against tmux on (a) per-workspace socket isolation parity, (b) pane control parity for the lead/teammate dispatch pattern, (c) Oz observability surface (Sub-Playbook C will have committed to a multiplexer-observation contract by then), (d) what "macOS-only" means for the CI matrix and for documented adopter requirements. Forward-compat hint for Sub-Playbook C: do not deepen Oz's tmux coupling beyond what observability requires; treat the multiplexer as an abstractable layer so this migration is tractable. |
